## 核心概念

### 1. Embedding（向量嵌入）

#### 什么是Embedding？
- 将文本转换为高维向量（数值数组）
- 相似的文本在向量空间中距离更近
- 捕捉文本的语义信息

#### 常用Embedding模型
| 模型 | 维度 | 语言 | 成本 | 适用场景 |
|------|------|------|------|----------|
| OpenAI ada-002 | 1536 | 多语言 | $0.0001/1K tokens | 通用 |
| sentence-transformers | 384-768 | 多语言 | 免费 | 本地部署 |
| text2vec | 768 | 中文 | 免费 | 中文任务 |
| m3e | 768 | 中文 | 免费 | 中文任务 |

#### 相似度计算
```python
# 余弦相似度（最常用）
similarity = dot(v1, v2) / (norm(v1) * norm(v2))

# 欧氏距离
distance = sqrt(sum((v1 - v2)^2))

# 点积
dot_product = sum(v1 * v2)
```

### 2. 向量数据库

#### 主流向量数据库对比

**Chroma**
- ✅ 优点：易用、自动持久化、支持metadata
- ❌ 缺点：性能一般、不适合大规模
- 💡 适用：开发测试、小型应用（<100K文档）

**FAISS**
- ✅ 优点：高性能、支持GPU、多种索引
- ❌ 缺点：API复杂、需手动管理
- 💡 适用：生产环境、大规模数据（>100K文档）

**Pinecone**
- ✅ 优点：完全托管、自动扩展、高可用
- ❌ 缺点：付费、数据在云端
- 💡 适用：商业项目、需要高可用性

**Milvus**
- ✅ 优点：企业级、高性能、功能丰富
- ❌ 缺点：部署复杂、学习曲线陡
- 💡 适用：大型企业、需要高级功能

#### 索引类型选择
```
数据量 < 10K    → Flat索引（精确搜索）
10K < 数据量 < 1M  → IVF索引（聚类索引）
数据量 > 1M     → HNSW索引（图索引）
```

### 3. 文档分块（Chunking）

#### 分块策略

**固定大小分块**
```python
chunk_size = 500  # tokens
chunk_overlap = 50  # 10%重叠
```

**按段落分块**
- 保持语义完整性
- 适合结构化文档

**递归分块**
- 优先按段落
- 段落太大则继续分割
- LangChain推荐方法

#### Chunk Size选择指南
| 文档类型 | 推荐大小 | 说明 |
|---------|---------|------|
| 问答对 | 200-300 | 短小精悍 |
| 文章段落 | 500-800 | 保持语义完整 |
| 技术文档 | 800-1200 | 包含完整概念 |
| 书籍章节 | 1000-1500 | 保持上下文 |
| 代码文件 | 300-500 | 保持函数完整 |

#### Overlap设置
- 推荐：10-20% 的chunk_size
- 作用：避免信息在边界处丢失
- 示例：chunk_size=500, overlap=50-100

### 4. RAG（检索增强生成）

#### RAG工作流程
```
用户问题
    ↓
Embedding（向量化）
    ↓
向量检索（相似度搜索）
    ↓
获取Top-K文档
    ↓
构建Prompt（问题+上下文）
    ↓
LLM生成答案
    ↓
返回结果
```

#### RAG核心参数
```python
# 推荐配置
chunk_size = 500        # 分块大小
chunk_overlap = 50      # 重叠大小
top_k = 3-5            # 检索文档数
temperature = 0         # 生成温度
similarity_threshold = 0.7  # 相似度阈值
```

#### RAG vs Fine-tuning
| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时 | 需要重新训练 |
| 成本 | 低 | 高 |
| 可解释性 | 高（可追溯来源） | 低 |
| 准确性 | 依赖检索质量 | 高 |
| 适用场景 | 知识密集型 | 任务特定型 |

### 5. 高级RAG技巧

#### Query改写
- 目的：优化查询以提高检索质量
- 方法：使用LLM改写用户查询
- 效果：+10-20% 准确率

#### Multi-Query
- 目的：从多角度检索
- 方法：生成3-5个相关查询
- 效果：提高召回率

#### HyDE（假设文档嵌入）
- 目的：缩小查询与文档的差异
- 方法：生成假设答案，用答案检索
- 效果：+15-25% 准确率

#### 重排序（Reranking）
- 目的：提高检索精度
- 方法：使用LLM对候选文档重新评分
- 效果：+15-30% 准确率
- 成本：较高（需要额外LLM调用）

#### 混合搜索
- 目的：结合语义和关键词
- 方法：向量搜索 + 关键词过滤
- 效果：提高精确度

#### 上下文压缩
- 目的：减少token消耗
- 方法：提取文档中与问题相关的部分
- 效果：-30-50% Token成本

### 6. RAG评估指标

#### 检索质量指标
- **Precision（精确率）**: 检索结果中相关文档的比例
- **Recall（召回率）**: 相关文档中被检索到的比例
- **F1 Score**: Precision和Recall的调和平均
- **MRR**: 第一个相关文档位置的倒数
- **NDCG**: 考虑位置的排序质量

#### 生成质量指标
- **Faithfulness（忠实度）**: 答案是否基于检索的文档
- **Answer Relevance**: 答案是否回答了问题
- **Context Relevance**: 检索的文档是否与问题相关

#### 系统性能指标
- **检索延迟**: 目标 <100ms
- **生成延迟**: 目标 <2s
- **总延迟**: 目标 <3s
- **吞吐量**: 目标 >10 QPS
- **成本**: 目标 <$0.01/query

## 实践经验

### 1. 向量数据库选择

**开发阶段**
- 使用Chroma
- 快速原型验证
- 易于调试

**生产环境**
- 小规模（<100K）：Chroma
- 中规模（100K-1M）：FAISS
- 大规模（>1M）：Milvus或Pinecone

### 2. 分块策略优化

**通用建议**
```python
# 默认配置
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "，", " ", ""]
)
```

**针对不同内容**
- 代码：保持函数完整，chunk_size=300-500
- 表格：按行分割，保留表头
- Markdown：按标题层级分割

### 3. 检索优化技巧

**提高精度**
1. 使用重排序
2. 添加metadata过滤
3. 调整相似度阈值
4. Query改写

**提高召回**
1. 增加top_k
2. 使用Multi-Query
3. 降低相似度阈值
4. 优化chunk策略

**平衡多样性**
1. 使用MMR检索
2. 调整lambda参数
3. 增加检索数量

### 4. Prompt工程

**基础模板**
```python
template = """基于以下上下文回答问题。如果上下文中没有答案，请说"我不知道"。

上下文:
{context}

问题: {question}

答案:"""
```

**高级模板**
```python
template = """你是一个AI助手。请基于提供的上下文回答问题。

要求:
1. 只使用上下文中的信息
2. 如果上下文中没有答案，明确说明
3. 提供引用来源
4. 回答要简洁明了

上下文:
{context}

问题: {question}

答案:"""
```

### 5. 成本控制

**降低Embedding成本**
- 使用开源模型（sentence-transformers）
- 批量生成embedding
- 缓存embedding结果
- 增量更新索引

**降低LLM成本**
- 使用gpt-4o-mini而不是gpt-4o
- 上下文压缩
- 减少检索文档数
- 实施缓存策略

**降低存储成本**
- 定期清理无用数据
- 使用压缩
- 选择合适的索引类型

### 6. 常见问题解决

**问题1：检索不到相关文档**
- 检查文档是否正确加载
- 调整chunk_size
- 增加top_k
- 检查embedding模型

**问题2：答案不准确**
- 优化Prompt模板
- 增加检索文档数量
- 改进文档质量
- 使用更强的LLM

**问题3：响应太慢**
- 优化向量数据库（使用FAISS）
- 减少检索文档数
- 使用缓存
- 考虑异步处理

**问题4：成本太高**
- 使用开源embedding
- 减少LLM调用
- 优化chunk数量
- 使用便宜的模型

# LangChain
langchain==0.1.0

### 工具函数
```python

from langchain_openai import OpenAIEmbeddings
embeddings = OpenAIEmbeddings()

# 文本分割
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)

# 向量存储
from langchain.vectorstores import Chroma
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings
)

# RAG链
from langchain.chains import RetrievalQA
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever()
)
```
